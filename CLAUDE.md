# lazy-sub2api — project context for Claude Code

懒猫微服 (LazyCat MicroServer) lpk wrapper for [sub2api](https://github.com/Wei-Shaw/sub2api).
Upstream sub2api is vendored as a git subtree at `vendor/sub2api/`. This repo
just contains caller-specific config + thin workflow wrappers — build/publish
logic lives in [microlazy-apps/lazycat-ci](https://github.com/microlazy-apps/lazycat-ci).

## Layout

```
lazy-sub2api/
├── vendor/sub2api/                   git subtree from Wei-Shaw/sub2api
├── lazycat/
│   ├── package.template.yml          committed; version: ${VERSION}
│   ├── lzc-manifest.template.yml     committed; main image: ${LAZYCAT_IMAGE}
│   ├── lzc-build.yml                 committed
│   ├── lzc-deploy-params.yml         committed; ADMIN_EMAIL / ADMIN_PASSWORD
│   ├── icon.png
│   ├── package.yml                   .gitignored, generated by build-lpk.sh
│   ├── lzc-manifest.yml              .gitignored, generated by build-lpk.sh
│   └── .gitignore                    *.lpk
└── .github/workflows/
    └── release.yml                   25 lines; calls lazycat-ci reusables
```

## Architecture: 3-service stack

sub2api needs three services:

| Service  | Image                                   | Purpose             |
|----------|-----------------------------------------|---------------------|
| main     | `${LAZYCAT_IMAGE}` (built by CI)        | Go backend + Vue UI |
| postgres | `registry.lazycat.cloud/lee/library/postgres:d5559a95f3e69bb8` | App DB |
| redis    | `registry.lazycat.cloud/lee/library/redis:ef2791ca3ea21111`    | Cache / sessions |

CI's `lpk-build.yml` only does ONE `lzc-cli appstore copy-image` (for the
main app image). Postgres/redis are pre-mirrored into the lazycat registry
out-of-band and pinned by digest in `lzc-manifest.template.yml`.

> ⚠️ The pinned `lee/library/...` URLs are temporary holdovers from the
> original publishing setup. To make this fully self-contained under the
> microlazy-apps namespace, run `lzc-cli appstore copy-image
> docker.io/library/postgres:18-alpine` (and redis equivalent) once and
> update the template with the resulting URLs.

## Template architecture

Caller commits `*.template.yml` files with `${VERSION}` / `${LAZYCAT_IMAGE}`
shell-style placeholders. lazycat-ci's `build-lpk.sh` renders them to
`lazycat/{package,lzc-manifest}.yml` (gitignored) via
`envsubst '${LAZYCAT_IMAGE} ${VERSION}'` — only those two vars are
substituted; lazycat's `{{ stable_secret }}` / `{{ .S.BoxDomain }}` /
`{{ .U.ADMIN_EMAIL }}` runtime templating is preserved.

## Release flow (`v*` tag)

```
git tag v0.1.124 && git push --tags
  ↓
release.yml (caller, this repo)
  ├── build:   uses lazycat-ci/lpk-build.yml@v0.1.0
  │             docker (ubuntu) → push ghcr (with v0.1.124 tag)
  │             build  (xiaomoru) → copy-image ghcr→lazycat →
  │                                  render templates → lzc-cli project build →
  │                                  artifact:lpk
  └── publish: uses lazycat-ci/publish.yml@v0.1.0
               publish-release (ubuntu)  → GH Release upload .lpk
               publish-appstore (xiaomoru) → lzc-cli appstore publish
```

## Versioning notes

- Upstream Wei-Shaw/sub2api uses `v0.x.y`.
- Original lazycat publishing (gitlab) used `v1.x.y` — last published: `v1.1.95`.
- This wrapper restarts versioning aligned with upstream tags. First release
  on this repo should bump above `1.1.95` (e.g. `v2.0.0`) to satisfy lazycat
  appstore monotonic-version requirement, OR use a fresh package id if a
  reset is desired.

## Deploy params

`lzc-deploy-params.yml` declares two install-time inputs surfaced by
lazycat as `{{ .U.ADMIN_EMAIL }}` / `{{ .U.ADMIN_PASSWORD }}`. These flow
into `main` service's environment as `ADMIN_EMAIL` / `ADMIN_PASSWORD`,
which sub2api's `AUTO_SETUP=true` consumes to bootstrap the first admin
user on initial container start.

## Lazycat manifest gotchas

- `command:` MUST be a string, not a YAML list. lpk-manager rejects array form.
- Quote any command containing `::` — YAML parses `:` + space as map separator.
- `version` MUST be strict semver. `lpk/read_from_cos` rejects bare SHAs.
- TCP ingress only routes on default subdomain — sub2api is HTTP-only so N/A here.

## One-time setup (org-level, already done if cloning sshx pattern)

- Org self-hosted runner `xiaomoru` registered at microlazy-apps level
- Org/repo secrets `LAZYCAT_USERNAME`, `LAZYCAT_PASSWORD`
- ghcr package `microlazy-apps/lazy-sub2api` visibility — defaults to PRIVATE
  on first push; flip to Public at
  https://github.com/orgs/microlazy-apps/packages/container/lazy-sub2api/settings
  before lazycat's `copy-image` can pull it
- Lazycat appstore registration of `cloud.lazycat.app.sub2api` —
  **already done** (current published version: v1.1.95). No bootstrap needed;
  release.yml takes over directly.

## Updating sub2api upstream

```sh
git subtree pull --prefix=vendor/sub2api \
  https://github.com/Wei-Shaw/sub2api.git main --squash
```

If sub2api adds new env vars or changes service contracts, update
`lzc-manifest.template.yml` accordingly before tagging a new release.

## Local lpk build (smoke test)

Requires `lzc-cli`, `envsubst` (gettext), and docker on PATH. Clone
lazycat-ci once and reuse its `build-lpk.sh`:

```sh
# one-time
git clone https://github.com/microlazy-apps/lazycat-ci ~/lazycat-ci

# in this repo:
docker build -t lazy-sub2api:dev ./vendor/sub2api
bash ~/lazycat-ci/scripts/build-lpk.sh \
  --image lazy-sub2api:dev --version 0.0.0-dev \
  --skip-docker-build
# → lazycat/cloud.lazycat.app.sub2api-v0.0.0-dev.lpk
```

For a CI-faithful build (pushes ghcr + lazycat registry), tag a `v*`
release and let `release.yml` run.
